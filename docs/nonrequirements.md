# 非機能要件

1. **デプロイ環境**
   - Azure Functions（Consumption Plan）を利用する。これによりインフラの自動スケーリングと従量課金を活かし、コストを抑えつつリクエスト増にも柔軟に対応する。
   - デプロイ先リージョンは Azure 東日本（japaneast）とする。

2. **実行プラットフォームと言語**
   - 開発言語は Python 3.11 とする。これにあわせて Function App のランタイム設定や依存管理（requirements.txt）も Python 3.11 に対応させる。

3. **パフォーマンスと運用**
   - 利用想定は低頻度アクセスで、レスポンスタイムへの厳密なSLAは設定しないが、実行結果がすぐに得られる程度の応答速度を目指す。
   - エラー発生時もHTTPステータスは200とするため、監視やテストでエラー判定をする際は本文を検査する必要がある点を運用手順に記載する。

4. **UX/インタフェース**
   - ブラウザでアクセスした際に結果が画面に表示されればよく、JSONやHTMLをレンダリングする必要はない。単純なプレーンテキストレスポンスで完了。

5. **その他制約**
   - 初期段階では特別なセキュリティ設定（認証や承認）は設けず、アクセス制限も想定しない。将来的な要件追加時には認証やロギングを検討する。

6. **将来の検討事項**
   - エラー本文のフォーマット統一（プレフィックスやコード）や、多言語対応の要否は今後の要件整理で検討する。
   - テスト自動化（例: Azure Functions の HTTP トリガーを使ったE2Eテスト）を後段で追加する可能性があることを記録しておく。
